# Goal
This document defines the metrics used to quantify the any-order inference of FlexMDM, and documents how `evals/tree_analysis` implements them.

# Assumption
We assume that the Python code generated by the model has already been converted into a tree. The tree can be some trimmed/pruned/coarse version of the AST. We also assume that we have access to the generation trace, i.e. the partial completion at each time step.

# Core definitions
1. We say a node is **non-leaf** if it has at least one child.
2. Notation: for each node `v`, let its children be `child(v) = [c1, ..., ck]`, where `k` is the number of children of `v`.
3. We say a node is a **split node** if it has at least two children. We say a node is a **sequence node** if it has exactly one child.
4. For a non-leaf node `v`, a **child block** of `v` is a subtree rooted at one of `v`'s children.
5. Each subtree has three time-dependent states:
   - **unstarted**: no part of the subtree has appeared yet,
   - **open**: some part of the subtree has appeared, but the subtree is not yet complete,
   - **finished**: the subtree has been fully generated.
6. A child is **unstarted/opened/finished** if the subtree rooted at this child is unstarted/opened/finished.

## Reporting convention
For each metric below, we recommend reporting two versions:
1. **Overall**: average over all non-leaf nodes, where sequence nodes contribute `1.0`.
2. **Split-only**: average only over split nodes.

The overall version answers: *did the generation order fit the inherent tree structure of this problem?*  
The split-only version answers: *when the tree actually offered branching opportunities, did the model exploit them?*

We also recommend reporting two tree variants when relevant:
1. **Code-only**: only executable algorithmic nodes.
2. **Reference-aware**: include docstring / examples / tests as separate top-level reference nodes.

# Metric 1: Coverage before commitment (CBC)
We first associate a number to each non-leaf node.

## Intuition
At a split node, this metric asks:
    Before the model fully completed one child block, how many of the sibling children blocks had it already opened?

This is the most natural scalar for "previewing multiple parts of the tree before committing."

## Local definition
Let `v` be a non-leaf node with children `[c1, ..., ck]`.
- If `k = 1`, define:
  - `CBC(v) = 1.0`
- If `k >= 2`, define:
  - `t_commit(v)` = the first time step at which **any** child block of `v` becomes finished.
  - `N_started(v)` = number of children blocks of `v` that have been started by time `t_commit(v)` (this includes the one that has just been completed)
Then define:
- `CBC(v) = N_started(v) / k`

## Interpretation
- `CBC(v) = 1/k` corresponds to strict serial behavior at this split node.
- `CBC(v)` close to `1` means that many children were already opened before the first one was completed.
- A high score means the model deferred local commitment and previewed multiple sibling blocks.

## Aggregate score
Let `V_nonleaf` be the set of all non-leaf nodes, and `V_split` the set of split nodes.
- **Overall Coverage-before-Commitment**
  - average of `CBC(v)` over `v in V_nonleaf`
- **Split-only Coverage-before-Commitment**
  - average of `CBC(v)` over `v in V_split`


# Metric 2: Return to unfinished blocks (RUB)

## Intuition
Coverage-before-commitment tells us whether the model opened multiple sibling nodes early.  
This second metric asks a stricter question:
    After leaving one child unfinished and moving to another child, does the model later come back?
This captures the "back-and-forth over different nodes" behavior that is closest to the intended notion of human-like any-order traversal.

## Child-level return condition
Fix a non-leaf node `v` with children `[c1, ..., ck]`.

A child `c` is said to satisfy the **return-to-unfinished-block** condition if:
between the time step at which the `c` is opened and the time step at which `c` is finished, the model still unmasks at least one token belonging to other sibling blocks.

In plain words: the model opened `c`, left it unfinished, worked on another child, and then came back.

## Local definition
Let `v` be a non-leaf node with children `[c1, ..., ck]`.
- If `k = 1`, define:
  - `RUB(v) = 1.0`
- If `k >= 2`, let `N_returned(v)` be the number of children of `v` that satisfy the return condition above.

Then define:
- `RUB(v) = N_returned(v) / k`

## Interpretation
- `RUB(v) = 0` means no genuine back-and-forth across unfinished sibling blocks.
- Higher `RUB(v)` means more revisiting of partially developed siblings.
- This is a stronger signal of genuine any-order node traversal than Metric 1.

## Aggregate score
- **Overall Return-to-Unfinished-Blocks**
  - average of `RUB(v)` over `v in V_nonleaf`
- **Split-only Return-to-Unfinished-Blocks**
  - average of `RUB(v)` over `v in V_split`


# Metric 2+: Graded return to unfinished blocks (RUB+)

## Motivation
The binary RUB above collapses two qualitatively different traces. Suppose `v`
has children `c1, c2`. In trace **A** the model writes one token of `c1`, one
token of `c2`, then finishes `c1`, then finishes `c2` (one switch each). In
trace **B** the model alternates between `c1` and `c2` many times before
finishing either. Both yield `RUB(v) = 1.0`: the metric saturates at the
*existence* of one return.

`RUB+` replaces that step indicator with a **three-level graded score** per
child: no return, exactly one return, or two-or-more returns. The cap at "two
returns" is intentional — going beyond two returns reflects frantic alternation
rather than genuine any-order, so we don't reward it further.

## Per-child runs
Fix a non-leaf node `v` with children `[c1, ..., ck]`. Restrict the trace to
entries that have at least one new-token in *some* child of `v` (entries
dominated by excluded subtrees, e.g. Generated Tests in `code_only`, are
ignored — leaving and re-entering `c` for an excluded sibling does not count
as a return, matching RUB's semantics). On this filtered trace, for each
child `c` of `v`, define
- `visits(c)` = number of maximal contiguous spans during which the model is
  unmasking tokens that belong to `c`'s subtree.

Equivalently, `visits(c) - 1` is the number of times the model left `c`,
worked on a sibling of `c` at `v`, and came back.

## Local definition
- If `k = 1`, define:
  - `RUB+(v) = 1.0` (sequence-node convention).
- If `k >= 2`, define per-child
  - `score(c) = min(visits(c) - 1, 2) / 2`
  so that
  - `visits(c) = 1` (no return) → `0.0`
  - `visits(c) = 2` (one return) → `0.5`
  - `visits(c) >= 3` (two or more returns) → `1.0`
  Children with zero revealed tokens contribute `0` to the sum. Then
  - `RUB+(v) = (1/k) * sum_c score(c)`

## Why no structural normalization
A natural worry is that very small children (e.g. a 2-token child of `v`)
cannot structurally exhibit two returns, so the cap-at-two formula penalizes
them. We measured this empirically on DC-base HumanEval traces
(a Dream-Coder HumanEval trace run at temperature 0.2, 164 samples, 1,229 children of split
nodes) and found:
- 88.9% of children of split nodes have enough tokens for two returns to be
  structurally possible (`min(n(c), Σ_{j≠c} n(j) + 1) >= 3`),
- 10.9% have exactly two tokens (one return possible, two not),
- 0.2% have a single token.
The structural correction would only affect the 11% tail and add complexity
without changing the headline numbers. We therefore keep the simpler
formulation above.

## Aggregate score
- **Overall RUB+**
  - average of `RUB+(v)` over `v in V_nonleaf`
- **Split-only RUB+**
  - average of `RUB+(v)` over `v in V_split`


# Metric 3: Open-block width

## Intuition
This metric asks:
    At the most parallel moment of generation for this node, how many sibling children were simultaneously open?

It is the most direct operational measure of how many parts of the same local tree region the model was actively keeping alive at once.

## Local definition
Let `v` be a non-leaf node with children `[c1, ..., ck]`.

At each time step `t`, let:
- `N_open(v, t)` = number of children of `v` that are open at time `t`

Define the local open-block width as:
- if `k = 1`, `OBW(v) = 1.0`
- if `k >= 2`, `OBW(v) = max_t N_open(v, t) / k`

## Interpretation
- `OBW(v) = 1/k` corresponds to strictly serial behavior at this split node.
- Higher `OBW(v)` means the model kept more sibling blocks simultaneously alive.
- This metric is especially easy to visualize on trace plots.

## Aggregate score
- **Overall Open-Block Width**
  - average of `OBW(v)` over `v in V_nonleaf`
- **Split-only Open-Block Width**
  - average of `OBW(v)` over `v in V_split`

# Implementation: how the extracted code is parsed into the metric tree

This section is a precise walkthrough of the exact transformation the code in
`evals/tree_analysis/{compute_metrics,visualize_final_ast_mapping}.py`
applies to a `.pt` trace before computing CBC / RUB / OBW. Every step name
below corresponds to a function in those two files.

## 0. Inputs

A single `.pt` record produced by `evals/humaneval_compare/generate.py`:

| field             | shape / type      | meaning                                                          |
|-------------------|-------------------|------------------------------------------------------------------|
| `prompt`          | `str`             | the raw prompt fed to the encoder                                |
| `prompt_len`      | `int`             | number of prompt-prefix tokens (boundary into `sequences[s]`)    |
| `sequences`       | `(S, L) int32`    | token state at each of S snapshots                               |
| `attention_masks` | `(S, L) bool`     | active positions per snapshot (1 = real token, 0 = pad)          |
| `insertion_masks` | `(S, L) bool`     | positions just inserted at step s (FlexMDM only; all-False for DC-base) |
| `mask_id`, `pad_id` | `int`           | tokenizer special ids                                            |
| `meta`            | `dict`            | `entry_point`, `task_id`, etc. — used by the `entry_only` variant |

## 1. Decode the body region into a string

`compute_metrics._decode_record_body` runs `tokenizer.decode([id])` on **each**
post-prompt position individually and concatenates the results to produce
`raw_decoded`. Per-position decoding (rather than `tokenizer.decode([all_ids])`)
gives us per-position character spans `position_char_spans[p] = (char_start,
char_end)` that we'll need at step 6 to align reveal-steps to Python tokens.
We verified empirically that the per-position concatenation matches the
batch-decoded string byte-for-byte for the tokenizer in use.

`raw_decoded` is then passed through `clean_generated_answer`, which simply
`.replace`'s the marker tokens (`<|beginoftext|>`, `<|endoftext|>`, the
visible-mask marker `[M]`) with empty strings and `.strip()`s leading /
trailing whitespace. Result: `cleaned_source`.

The offset of `cleaned_source` inside `raw_decoded` (`raw_to_clean_offset =
raw_decoded.find(cleaned_source)`) is recorded so step 6 can translate
between coordinate systems.

## 2. Parse with fallbacks (`parse_with_fallbacks`)

Tries to obtain an `ast.Module` by attempting four different sources of
Python text in order. The first one that successfully parses wins, and its
name is recorded in `parse_source_kind`:

1. **`cleaned_answer`** — `ast.parse(cleaned_source)` directly. Wins when the
   model emitted runnable Python without trailing markdown / prose.
2. **`fenced_code_block`** — `extract_fenced_python` regex-extracts the
   contents of the first ```` ```python ... ``` ```` block (case-insensitive,
   `(?:python|py)?` opener). Wins when the model wraps its answer in a
   markdown code block.
3. **`before_closing_fence`** — `cleaned_source.split("```", 1)[0].strip()`.
   Wins when the prompt already opened a code fence (DC instruct) and the
   model only emitted a *closing* fence followed by natural-language prose.
4. **`longest_valid_prefix`** — `extract_longest_valid_code` (vendored in
   `evals/humaneval_compare/_sanitize_utils.py`). Picks the contiguous line range
   `[i:j]` (with i, j ≤ 100) whose join is the longest syntactically-valid
   Python snippet. Wins when generation was truncated mid-statement (e.g.
   `max_new_tokens` cut a trailing assert).

The winning text is stored as `parse_source` (this is what every later step
sees as "the source"). If all four fail, `tree=None`, the sample is recorded
as `parse_success=False`, and an empty metrics payload is emitted.

## 3. AST → record tree (`ast_to_record_tree`)

A pre-order DFS over the `ast.Module` that:
- assigns each visited AST node an integer id (its index in the `records`
  list, which is the order of first visit),
- records its `type` (Python AST class name), a human-readable `label`, the
  parent's id, the list of children ids, and the depth (root = 0),
- computes `char_start` / `char_end` in `parse_source` from
  `lineno` / `col_offset` / `end_lineno` / `end_col_offset` (with a special
  case for the `Module` root, which gets `char_start=0`,
  `char_end=len(parse_source)`).

The output is **the full Python AST** (every `ast.AST` subclass gets a
record), `ast_obj_to_id` (mapping `id(ast_node) → record_id`), and
`ast_parent` (`record_id → parent_record_id`). This is **not** the metric
tree — it's the raw scaffold the next step consumes.

## 4. AST → visualization tree (`build_visualization_tree`)

Produces the simplified "viz tree" that the metrics actually run over. The
viz tree is much smaller than the AST tree because it keeps only
**statement-level** structure and applies two semantic collapses:

### 4a. Allowed node types

Only AST classes in `VISUAL_STATEMENT_TYPES` become viz nodes:
```
FunctionDef AsyncFunctionDef ClassDef
For AsyncFor While If Try With AsyncWith Match
Assign AnnAssign AugAssign Return Expr
Import ImportFrom Raise Assert Break Continue Pass Delete
Global Nonlocal
```
Expression-level AST nodes (`BinOp`, `Name`, `Call`, `arguments`, …) **do
not** become viz nodes; they live only in the underlying record tree.

### 4b. Recursion through statement bodies (`process_body` / `process_statement`)

Starting at `Module`, for each statement in the body:

- If the *first* body statement is an `Expr(Constant(str))` (i.e., a
  docstring), it is **collapsed**: no viz node is created for it; the owner
  function/class/module gets `docstring_collapsed=True`. The docstring's AST
  records still exist; they're remapped via `ast_to_viz` to the owner's viz
  node so any tokens inside the docstring will count toward the owner.

- Compound statements recurse into their bodies:
  - `FunctionDef` / `AsyncFunctionDef` / `ClassDef` → recurse via
    `process_body` (so their docstrings collapse into them).
  - `For` / `AsyncFor` / `While` / `If` → recurse into `body` *and* `orelse`,
    skipping any docstring-shaped child.
  - `With` / `AsyncWith` → recurse into `body` only.
  - `Try` → recurse into `body`, every `handler.body`, `orelse`,
    `finalbody`.
  - `Match` → recurse into every `case.body`.
- Simple statements (`Assign`, `Return`, `Expr`, etc.) are leaves of the viz
  tree.

### 4c. "Generated Tests / Examples" detection (`detect_generated_test_start`)

At the top level (Module), after we've seen a `FunctionDef` / `ClassDef`,
if the next statement is something model-trailing-ish — `Assert`, top-level
`Call` (e.g. `print(f(1))`), or `if __name__ == "__main__":` — then **all
remaining top-level statements** are wrapped into a single synthetic viz
node:

```
type:  "Generated Tests"
label: "Generated Tests / Examples"
synthetic: True
```

Its `char_span` covers the union of all wrapped statements. The wrapped
statements' AST records are *remapped* via `ast_to_viz` to this synthetic
node (so their tokens still count, but they have no individual viz nodes
under it). The detection reasons (e.g. "top-level call/print after function
/ class") are recorded for diagnostics.

This is the mechanism by which the `code_only` variant later excludes
trailing test boilerplate without dropping the underlying AST data.

### 4d. Output

`viz_nodes` is a list of dicts with `viz_node_id`, `type`, `label`,
`parent_id`, `children`, `depth`, `ast_node_id` (or `ast_node_ids` for the
synthetic Generated-Tests node), `char_start` / `char_end`, and an empty
`token_indices` placeholder filled at step 5.

`ast_to_viz` maps each AST record id to the viz node id whose subtree
contains it (via either direct creation in 4b or remapping in 4a / 4c).

## 5. Tokenize the source (`tokenize_source`)

Uses the standard library `tokenize.generate_tokens(io.StringIO(parse_source).readline)`.
We drop `ENCODING` and `ENDMARKER` pseudo-tokens and convert every remaining
token (NAME, NUMBER, STRING, OP, NEWLINE, INDENT, DEDENT, etc.) into a dict:

| field         | meaning                                              |
|---------------|------------------------------------------------------|
| `token_index` | position in the emitted token stream (0-based)       |
| `string`      | the literal text                                     |
| `type`        | std-lib `token.tok_name[t.type]`                     |
| `start_line`/`start_col`/`end_line`/`end_col` | `tok.start`, `tok.end` |
| `char_start`/`char_end` | byte offsets into `parse_source` (via `position_to_offset`) |

`TokenError` / `IndentationError` from `generate_tokens` are caught and the
partial token list (those produced before the error) is returned; the error
message is recorded as `tokenize_error`. The metric still runs, just on the
prefix.

## 6. Map tokens → viz nodes (`map_tokens_to_nodes`)

For each Python token:

1. Find every AST record (excluding `Module`) whose `[char_start, char_end]`
   intersects the token's `[char_start, char_end]`. Intersection is strict
   (`max(starts) < min(ends)` for non-empty tokens) with one special case
   for zero-width tokens (must lie inside the node's range).
2. Among intersecting records, pick the **deepest** (`depth` max), tiebreak
   by **smallest span** (so the most-specific containing node wins). Call
   that the token's `deep_node_id`.
3. Walk up the `ast_parent` chain from `deep_node_id` until we hit an entry
   in `ast_to_viz`. That viz node is the token's `viz_node_id`. (Most
   AST nodes don't have a viz node directly — only the visual-statement
   types do — so this walk-up is the typical case.)

Each viz node accumulates two lists:
- `direct_token_indices`: tokens whose deep AST owner is *inside* the viz
  node's own AST subtree (no descendant viz node claims them).
- `subtree_token_indices`: the transitive closure (also includes tokens
  that landed in descendant viz nodes).

The metric uses `subtree_token_indices` as the canonical "this many tokens
belong to this viz node's subtree".

## 7. Per-position reveal step (`_reveal_step_per_final_position`)

This step takes the `.pt` trace and, for **each position p in the final
sequence**, computes the earliest step at which the token at that final
position was last set to a non-mask value.

For DC-base traces (no insertions): trivial — position p in the final
sequence is also position p in every snapshot. Reveal step is the smallest
`s` such that `sequences[s, p] != mask_id`.

For FlexMDM traces (with insertions): position indices shift over time. We
walk the trace **backward** with `insertion_masks` to build a lineage map
`final_positions_per_step[s]: List[int]`. The contract from
`flexmdm.inference._apply_after_token_insertions` is:

> at step s+1, positions where `insertion_masks[s+1]` is True are
> freshly-inserted (no parent in step s); the remaining active positions in
> step s+1 correspond *in order* to the active positions in step s.

So we drop the inserted entries from `lineage[s+1]` to obtain `lineage[s]`.
Then for each final-step position `p_final`, we walk the lineages and find
the smallest `s` whose mapped position has a non-mask token there. Final
fallback = the last step (i.e., this position was never seen non-mask
before it was committed at the final step).

## 8. Per-Python-token reveal step (`_python_token_reveal_steps`)

Each Python token's char span (in `parse_source`) needs translation back
to the `.pt` decoded coordinate system. Since `parse_source` is either
`cleaned_source` (cases 1, 2 of step 2) or a substring of `raw_decoded`
(cases 3, 4), we add `raw_to_clean_offset` to translate parse-source spans
into raw-decoded spans, then look up the per-position char spans recorded
in step 1 to find which `.pt` body positions overlap.

A Python token's reveal step is then the **max** over the reveal steps of
all overlapping `.pt` body positions. This conservatism — a Python token is
"revealed" only when *every* underlying byte-position is settled.

## 9. Generation progress (`build_generation_progress`)

We collect the `countable_token_indices` (tokens with positive char width)
and group them by their reveal step. For each *unique* reveal step (sorted
ascending) we emit one `entry` containing:

- `progress_index`: 0, 1, 2, …
- `step`: the actual snapshot index `s ∈ [0, S)`
- `progress`: `step / max(1, S - 1)`
- `new_token_indices`: tokens whose reveal step equals this entry's step
- `generated_token_indices`: cumulative tokens revealed so far
- `viz_completion[str(viz_node_id)]`: per-viz-node dict
  `{generated, new, total, ratio}`, where `total = |subtree_token_indices ∩
  countable|`, `new = |new_tokens ∩ subtree|`, `generated = |cumulative
  ∩ subtree|`.

Note: only steps that actually revealed a Python token produce an entry —
we don't emit an entry per snapshot.

## 10. Compute metrics per variant (`compute_tree_order_metrics_variant`)

Three variants are produced (see `compute_metrics.METRIC_VARIANTS`):

- **reference_aware** — every non-leaf viz node, including the synthetic
  `Generated Tests` node and its ast-remapped children.
- **code_only** — start a stack from every viz node whose type is in
  `REFERENCE_NODE_TYPES = {"Generated Tests", "Reference/Test Cases",
  "Generated Tests / Examples", "Examples"}`, then walk each one's
  `children` recursively to build an `excluded` set. Any viz node in this
  set (or whose parent is in it) is skipped during metric computation.
- **entry_only** — find the AST `FunctionDef` / `AsyncFunctionDef` whose
  `name == record["meta"]["entry_point"]`, map it to its viz node id, then
  filter `viz_nodes` to that node and its descendants. Drops the surrounding
  `Module` / `ImportFrom` / trailing-prints / `__main__`-block siblings.

For each non-excluded non-leaf viz node `v` with `k = len(children)`:

- **k = 1 (sequence node)**: by convention from §"Reporting convention"
  above, `CBC = RUB = RUB+ = OBW = 1.0`. (`is_split = False` is recorded.)
  We do also record `started_at_commit_count` and friends as `1 if total > 0
  else 0` for diagnostics.

- **k ≥ 2 (split node)**: with `totals[c] = total tokens in c's subtree`
  computed once,
  - `t_commit` = the first entry where some child has `generated >= total`
    (i.e., a child becomes fully revealed).
  - `started_at_commit_count` = number of children with `generated > 0` at
    that entry.
  - **CBC(v) = started_at_commit_count / k**.
  - For each child c with `total > 0`: find its `open_index` (first entry
    where `generated > 0`) and `finish_index` (first entry where `generated
    >= total`). If between them there's any entry where some *other* child's
    `new` count > 0, c counts as "returned to". `RUB(v) = returned / k`.
  - **RUB+**: in the same pass, for each child c we count
    `visits(c)` = the number of maximal contiguous runs of *entries with at
    least one new-token in some non-excluded child of v* in which
    `new(c) > 0`. (Entries where every non-excluded child of v has
    `new == 0` are skipped — they don't break a run.) Then
    `score(c) = min(visits(c) - 1, 2) / 2 ∈ {0, 0.5, 1.0}`
    (children with `total = 0` contribute 0), and
    `RUB+(v) = (1/k) * Σ_c score(c)`. Per-node, we also store
    `child_visit_counts` (the raw `visits(c)` per child) for diagnostics.
  - For each entry, count children with `0 < generated < total` (i.e.,
    *open*). Track the running max across entries. **OBW(v) = max_open / k**.
    Special case: if max_open is 0 throughout but at least one child was
    ever generated, set max_open to 1.

`is_split = True` is recorded; the entry also carries the raw counts for
debug.

## 11. Aggregate (`aggregate_tree_order_metrics`)

Given `local_metrics` (one entry per non-leaf node visited above):

- `nonleaf_node_count`, `split_node_count` — counts.
- `overall.{cbc,rub,obw}` — mean over all entries (sequence-node
  contributions of 1.0 included; split contributions as computed).
- `split_only.{cbc,rub,obw}` — mean over entries with `is_split=True`.
- `by_depth[d].{node_count, split_node_count, cbc, rub, obw}` — the
  same, bucketed by node depth.

The CLI's `summary.json` emits these fields per (gen_dataset, model, alg)
key, separately under each of the three variants.

## What this means in practice

- **The metric "tree" is the simplified visualization tree**, not the raw
  Python AST. Expression-level nodes (`Call`, `BinOp`, `Name`, …) are
  invisible to the metric — they only enter via the tokens that get
  attributed to their nearest containing statement.
- **Docstrings collapse upward**: a function with `def f(): """doc"""\n  body`
  has the docstring's tokens counted as belonging to the FunctionDef viz
  node directly, not as a separate child. So the FunctionDef may be
  classified as having one fewer child than its naive AST representation.
- **Trailing model boilerplate** (top-level `assert` / `print` / `__main__`
  blocks after a function/class) is grouped into one synthetic
  `Generated Tests` node by default. The `code_only` variant removes that
  whole subtree; `reference_aware` keeps it.
- **The structural floor on `overall_*`**: every sequence (k=1) viz node
  contributes the convention value 1.0. On HumanEval, ~57 % of non-leaf
  viz nodes are sequence nodes (Module-level chains, control-flow that
  has only one branch, etc.), so `overall_*` numbers are pulled toward 1.0
  irrespective of model behavior. **`split_only_*` is the informative
  column** for comparing models.
- **`entry_only` removes most of the structural-1.0 dilution** by dropping
  the Module / Import / trailing-tests siblings, but on the runs we've
  measured (DC-base on HE, both temperatures), it does not materially
